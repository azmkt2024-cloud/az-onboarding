# 보상 Q&A 쉬운 말 재작성 가이드 (스타일 스펙)

> 크롤링한 키움 보상지원센터 상담사례(어려운 약관 용어)를 **신입 설계사·일반인도 이해하기 쉬운 말 + 가상 예시**로 재작성하는 기준 문서입니다.
> **현재 200건은 이 기준으로 재작성 완료. 향후 나머지 2,848건도 반드시 이 가이드를 그대로 적용해 일관성을 유지합니다.**

---

## 0. 절대 원칙 (사실 보존)

1. **숫자·분류·기간·금액·조건은 원문 그대로 유지한다.**
   - 예: `3종 수술비`, `90일 면책`, `청약일로부터 5년`, `자기부담금 20%`, 분류표상 종(種) 번호, 금액 등은 한 글자도 바꾸지 않는다.
2. **핵심 결론(보상 가능/불가, 조건)을 바꾸지 않는다.** 문체와 예시만 손본다.
3. 애매하거나 원문에서 결론이 불명확하면, **원문의 표현을 보존**하고 단정적으로 바꾸지 않는다.
4. 광고/면책 보일러플레이트(상담 신청, 유의사항 안내문 등)는 **넣지 않는다.**
5. `category`와 `id`는 **변경 금지.**

---

## 1. 질문(question) 작성

- 실제 상담자가 묻듯 **자연스럽고 쉬운 1~2문장.**
- 어려운 약관 용어는 풀어 쓰되, 필요하면 괄호로 원어 병기: `실효(보험 효력 정지)`.
- 원문 질문의 의도를 유지한다.

## 2. 답변(answer) 작성 — 구조

1. **한 줄 결론 먼저** (예: "실효의 종류에 따라 달라집니다.")
2. **가상 예시 1개** — `예) 김○○님이 …` 형태. 사실을 바꾸지 않는 범위에서 이해를 돕는 상황만 만든다. (가상 인물명은 `○○` 마스킹)
3. **숫자·분류·조건은 원문 그대로**, 필요하면 불릿(•)으로 정리.
4. 마지막에 **`핵심:`** 한 줄 또는 짧은 주의.
- 분량: **4~8줄 내외.** 과도하게 길지 않게.
- 줄바꿈은 `\n` 사용 (불릿은 `• `).

---

## 3. 예시 (Before → After)

**[지급기준] 원문**
- Q: 실효 중에 발생한 보험사고의 경우, 보상가능한가요?
- A: 어떤 실효상태인지 따라서 보상이 달라요. 1. 특별부활(계속 보험료 1회 내지 않는 미납 상태)해당되는 경우에는 콜센터 통해서 보험료 입금 후, 보상 가능합니다. 2. …

**→ 재작성**
- Q: 보험료를 못 내서 보험이 '실효'(효력 정지)된 사이에 사고가 났어요. 보상받을 수 있나요?
- A:
  ```
  실효의 '종류'에 따라 달라집니다.
  • 예) 박○○님이 이번 달 보험료 1회를 깜빡해 미납한 상태(=특별부활 대상)라면 → 콜센터에 전화해 밀린 보험료를 입금하면 그 사고도 보상받을 수 있는 경우가 많습니다.
  • 오래 미납해 완전히 실효된 경우 등은 처리가 다르니 콜센터/담당자 확인이 필요합니다.
  핵심: '1회 미납(특별부활)' 상태면 보험료 입금 후 보상 가능성이 있습니다.
  ```

**[수술비] 주의** — "몇 종 수술비"처럼 **분류표 종(種) 번호는 반드시 원문 그대로** 두고, 앞에 쉬운 설명·예시만 덧붙인다. 숫자/분류를 요약·생략하지 않는다.

---

## 4. 출력 형식 (배치 작업 시)

각 항목을 아래 JSON 객체로 출력하고, 전체를 JSON 배열로 저장한다:

```json
{ "id": 1, "category": "지급기준", "question": "쉬운 질문…", "answer": "쉬운 답변…\n• …\n핵심: …" }
```

- `id`, `category`는 원본과 동일하게 유지.
- 200건 → `lib/data/compensationQnaEasy.ts`, 추후 전체 → 같은 형식으로 교체.

---

## 5. 재현 파이프라인 (향후 2,848건도 동일하게)

전체 3,048건도 **아래 순서를 그대로** 돌리면 같은 품질·일관성으로 만들 수 있습니다.

1. **크롤링** — `키움보상질문_크롤링.py`를 전체 페이지로 실행 → `키움_보상질문_*.json` 생성.
2. **청크 분할** — `node scripts/split_qna_chunks.mjs` (입력 JSON 경로만 새 파일로 교체) → `scripts/_qna_src/chunk_*.json` (30건 단위).
3. **재작성** — 각 청크를 **이 가이드(0~4장)를 시스템 프롬프트로** 사용해 재작성 → `scripts/_qna_out/chunk_*.json`.
   - 방법 A: 본 작업처럼 **병렬 서브에이전트**(청크당 1 에이전트).
   - 방법 B: **오프라인 LLM 배치**(이 가이드를 프롬프트로 고정해 API 호출). 3,048건은 B가 효율적.
4. **조립 + 자동 검증** — `node scripts/assemble_qna_easy.mjs`:
   - 건수/카테고리/id 일치, 중복 검사.
   - **숫자 토큰(종·일·년·개월·%·만원·회·세) 보존 자동 점검** → 누락 의심 항목을 `scripts/_qna_factcheck.json`에 기록.
5. **검수** — `scripts/_qna_factcheck.json`에 잡힌 항목만 원문과 대조해 보정(`scripts/patch_qna.mjs` 형태로 수정 후 4번 재실행). ← *담당자 검수 단계.*
6. **반영** — `lib/data/compensationQnaEasy.ts`(앱 fallback)와 `supabase/seed_qna.sql`(DB seed) 자동 생성 → Supabase에 seed 실행.

> 핵심: **가이드(이 문서)를 프롬프트로 고정** + **숫자 보존 자동검증 + 검수**가 일관성의 두 축입니다.

### 5-1. 오프라인 LLM 배치 (방법 B) — 3,048건 자동 처리

`scripts/rewrite_qna_batch.mjs` 가 위 1~4단계(크롤 파싱 → 재작성 → 검증 → 파일생성)를 **Anthropic Message Batches API**로 한 번에 수행합니다. 이 가이드 문서를 그대로 system 프롬프트로 고정(+프롬프트 캐싱)하므로 일관성이 보장됩니다.

```bash
# 0) 키 설정 (.env.local 또는 셸). 오프라인 전용 — 런타임 앱은 API를 쓰지 않음.
export ANTHROPIC_API_KEY=sk-ant-...

# 1) 미리보기 (과금 없음): 파싱 건수·샘플 요청 확인
node scripts/rewrite_qna_batch.mjs 키움_보상질문_전체.json

# 2) 실제 제출 (배치 생성 → 폴링 → 파일 생성)
node scripts/rewrite_qna_batch.mjs 키움_보상질문_전체.json --go

# 3) 비용 절감 시 모델 변경 (기본 claude-opus-4-8)
QNA_MODEL=claude-sonnet-4-6 node scripts/rewrite_qna_batch.mjs 키움_보상질문_전체.json --go
```

산출물: `lib/data/compensationQnaEasy.ts`(앱 fallback) + `supabase/seed_qna.sql`(DB seed) + `scripts/_qna_factcheck.json`(숫자 누락 의심 → **검수 대상**).
- 배치는 표준가 **50% 할인**, 보통 1시간 내 완료(최대 24h).
- 완료 후 `scripts/_qna_factcheck.json` 항목만 원문과 대조해 검수 → 필요 시 보정 후 seed 실행.
